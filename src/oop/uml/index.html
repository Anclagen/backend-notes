Unified Modeling Language

Unified Modeling Language (UML) is a standardized general-purpose modeling language in software engineering. It is widely used to visualize the design of a system and to communicate the system's design to others. UML provides a standard way to develop diagrams that visualize the design of a system.

UML Diagram Tools

Structural UML Diagrams

  • Class Diagram: A static structure diagram that describes the system's classes, attributes, operations (methods), and object relationships. It is the main building block of object-oriented modeling, used for both conceptual modeling and detailed modeling that translates into programming code.
  • Package Diagram: A static structure diagram that shows the system's packages, their classes, and object relationships. It organizes classes into packages and shows dependencies between them.
  • Object Diagram: A static structure diagram that depicts the system's objects, attributes, operations (methods), and relationships at a specific point in time. It provides a snapshot of the objects and their relationships.
  • Component Diagram: A static structure diagram that represents the system's components, their interfaces, and relationships. It illustrates the physical components and their dependencies.
  • Composite Structure Diagram: A static structure diagram that shows the system's composite structures, their parts, and relationships. It details the internal structure of a class and the relationships between its parts.
  • Deployment Diagram: A static structure diagram that describes the system's nodes, components, and their relationships. It shows the physical deployment and node dependencies.

Behavioral UML Diagrams

  • Use Case Diagram: A behavioral diagram created from use case analysis, showing relationships among actors and use cases. It helps identify, clarify, and organize system requirements.
  • Activity Diagram: A behavioral diagram that depicts the flow of control or objects in a system, similar to a flowchart. It shows the sequence of activities involved in a scenario.
  • Sequence Diagram: An interaction diagram showing object interactions arranged in time sequence. It illustrates the sequence of messages exchanged between objects to carry out a scenario's functionality.
  • State Diagram: A behavioral diagram that shows the flow of control from one activity to another. It depicts the overall flow and sequence of activities in a scenario.
  • Communication Diagram: An interaction diagram that shows how objects interact and in what order, arranged in time sequence. It details the objects, classes, and message exchanges needed for a scenario.
  • Interaction Overview Diagram: An interaction diagram that combines elements of activity and sequence diagrams, showing the overall flow and sequence of interactions in a scenario.
  • Timing Diagram: An interaction diagram that illustrates object interactions and time constraints. It shows the sequence of messages and interactions over time.

UML Relationship Notation

UML uses symbols to represent the visibility of class members and relationships between classes. The most common visibility symbols are:

  • Public (+): Visible to all classes.
  • Private (-): Visible only within the class itself.
  • Protected (#): Visible within the class and its subclasses.
  • Package (~): Visible to classes within the same package.

Lines in UML diagrams denote relationships between components. The standard notation for these lines is as follows:

  • Association: A solid line connecting two classes, representing a relationship between them.
  • Aggregation: A hollow diamond at the end of a line, representing a whole-part relationship.
  • Composition: A filled diamond at the end of a line, representing a strong whole-part relationship where the part cannot exist without the whole.
  • Generalization: A solid line with a hollow triangle at the end, representing an inheritance relationship between a superclass and a subclass.
  • Dependency: A dashed line with an arrow, representing a dependency relationship where one class depends on another.
  • Realization: A dashed line with a hollow triangle, representing an implementation relationship between an interface and a class that implements it.
